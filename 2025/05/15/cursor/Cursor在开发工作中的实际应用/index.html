Cursor在开发工作中的实际应用

发表于 2025-05-15 分类于 Cursor系列 阅读次数：

一、引言

很高兴与大家分享现阶段 Cursor 在我的工作中的使用体验。首先是预期管理，本篇文章不会分享 x 个你可能不知道的小技巧，也不会让你拥有无需自行编码的能力，同时不涉及 Cursor 工程化方面内容。仅仅是围绕个人开发流程中的已有问题，分享如何使用 Cursor 来提升这部分的开发体验，在工作中持续保持好的节奏和状态。

文章要点

  • 明确 Cursor 定位：不是开发者的替代品，而是放大开发者能力的工具
  • 分析使用误区：直接根据 PRD 生成代码存在两类典型问题
  • 重构开发流程：将开发分为规划阶段和执行阶段，明确 Cursor 在不同阶段的角色
  • 建立协作机制：通过反馈循环实现开发者与 Cursor 的有效协作
  • 应用场景总结：在现状分析、方案设计、影响评估三大场景中的实际应用

二、就差一个程序员了

最近团队在大力推广 Cursor AI，随着几个版本迭代体验下来，其精准的自动补全深得我心，具体可以体现在 Tab 键的使用率已逐渐高于 Command + C/V。既然这么懂我，那么能否更进一步，根据 PRD 直接输出代码呢？

2.1 从需求到代码

Cursor 能够理解代码上下文，从而根据简短的描述生成符合上下文的代码，于是我尝试直接将 PRD 提供给 Cursor 来生成代码：

几个需求尝试下来，总的来说分为两类问题：

问题一：知道 AI 不知道

描述： 你知道 AI 不知道的东西，但你仍然期望它能给出完美答案

示例： 比如直接让 Cursor 写个功能点，但没提供参考项、接口协议和交互方式

分析： 当我们把需求直接丢给 Cursor 时，常常会发现：

  • 输出质量低： 不符合项目的编码规范和架构设计
  • 不符合期望： 解决方案不是我们心中所想的
  • 需要频繁修改： 比起从头开始，修改和适配 AI 的代码可能更费时间

这种情况下，我们明知 AI 缺乏必要的上下文，却仍然期望它能理解我们的意图。结果常常是：大部分时间用在了修正 AI 的输出上，而非解决实际问题。

问题二：不知道 AI 不知道

描述： 更危险的是，你不知道 AI 不知道什么

示例： 比如直接将 PRD 贴给 Cursor，自己却对代码实现现状、需求预期和设计方案没有足够的了解

分析： 当我们对自己的需求和代码现状理解不深时，我们无法有效地：

  • 审查 AI 输出： 是否真的符合业务需求？
  • 评估实现质量： 是否会引入新的问题？
  • 判断边界情况： 是否考虑了所有可能的情况

这就像你去理发店，希望 Tony 老师稍微剪短一点，结果却被剪得稍微短了点。而这需要我们在开始之前对齐认知，补充描述和参照。在这个前置阶段，即使发现对方理解有偏差，也还能及时纠正。俗称“对齐颗粒度”。

2.2 从规划到执行

Cursor 产出的代码由它所接收的上下文决定，如果没有准确描述需求意图，它会通过推断做出假设，导致产出不准确。因此我们在使用 Cursor 时，关键在于区分开发过程中的规划阶段和执行阶段。

Cursor 在这个过程中，不应该被视为开发者的替代品，而是一面能够放大开发者能力的镜子：

  • 对于已知的部分，Cursor 可以加速实现，减少重复劳动。
  • 对于未知的部分，Cursor 可以协助探索，但不能替代开发者的判断。

在理解了 AI 的角色后，我们需要重构目前的开发工作流，让 AI 成为真正有效的助手。最关键的转变是：不再试图让 AI 替代开发流程中的任何环节，而是让它协助完成每个环节。这意味着不是把 PRD 扔给 AI，等待完整代码，而是和 AI 一起理解 PRD 和代码现状，共同设计方案，明确步骤，然后分步实现。

三、总结与展望

3.1 开发阶段拆分与 Cursor 角色定位

基于前面的分析，我们将开发流程分为两个阶段，明确 Cursor 在不同阶段的角色：

分析代码现状和约束

3.2 开发痛点分析与 Cursor 缓解作用

前端开发中方案设计存在效率与质量的矛盾：

  • 传统方法：模板化、组件库等，但缺乏针对性
  • Cursor 缓解：
    • 自动化分析代码结构和业务逻辑
    • 生成可视化成果（如时序图、流程图）
    • 减少重复劳动，提升方案质量

3.3 协作流程：反馈循环

核心是建立”反馈循环”机制：

  1. 开发者补充上下文：提供业务背景、技术约束、项目规范等
  2. Cursor 生成内容：基于上下文生成代码或方案
  3. 开发者审核反馈：检查质量、提出修改建议
  4. 循环优化：持续迭代直到满足要求

分为两个子循环：

  • 规划阶段：方案审核循环，确保设计合理
  • 执行阶段：代码审核循环，确保实现正确

3.4 Cursor 在三大场景的应用

现状分析

通过精心设计的提示词，让 Cursor 输出：

  • 业务功能脉络
  • 代码结构分析
  • 字段依赖关系
  • 模块间交互逻辑

帮助开发者快速了解陌生业务模块，降低学习成本。

改动方案

引导 Cursor 采用”先分析问题，再设计方案”的流程：

  • 分析现有代码结构和业务逻辑
  • 识别改动点和影响范围
  • 拆解复杂任务为可管理的模块
  • 确保方案贴合业务需求与架构设计

影响评估

借助 @git 指令，让 Cursor 分析代码改动：

  • 识别潜在风险点
  • 生成自测用例
  • 评估回归测试范围
  • 提升自测可信度和覆盖率

3.5 AI 时代开发者能力转型

AI 降低了信息获取门槛，开发者核心能力需要从”记忆与执行”转向”提出好问题”：

  • 积累专业知识：深入理解业务领域和技术栈
  • 提升提问能力：学会如何与 AI 有效沟通
  • 利用工具聚焦核心：让 AI 处理重复性工作，专注创造性任务
  • 实现双赢：个人成长与工作成果并重

3.6 展望

随着 AI 技术的不断发展，开发者需要：

  • 持续学习新的 AI 工具和最佳实践
  • 培养与 AI 协作的思维模式
  • 在保持技术深度的同时，提升跨领域协作能力
  • 将 AI 作为提升效率的伙伴，而非替代品
